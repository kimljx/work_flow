## 部门任务协同系统 V1 前后端分离详细执行计划（Windows 主部署，兼容 Win7/Linux）

### Summary
- 将现有方案重构为**单仓前后端分离**架构：`frontend/` + `backend/` + `deploy/` + `docs/`。
- 保留既定业务规则：任务域/通知域解耦、邮件驱动状态、QAX 仅通知展示、延期审批并发原子生效、模板匹配确定性、重要操作二次确认。
- 新增管理员专属“**用户与角色管理**”，支持新增成员/管理员、编辑、禁用、重置密码。
- 部署目标改为：**Windows 单机为主生产形态**，同时兼容 **Windows 7** 和 **Linux**；提供 `venv+脚本`、`Docker`、`PyInstaller` 三种交付方式；支持**内网离线部署**。
- 技术定版：
  - 前端：`Vue3 + Vite`，构建目标 `ES2019`
  - 后端：`FastAPI + SQLAlchemy + Alembic`，兼容 `Python 3.8`
  - 鉴权：`JWT + Refresh Token`
  - 接口前缀：`/api/v1/*`
  - 数据库：`SQLite`

### Implementation Changes
#### 1. 仓库与目录结构
- 单仓重建，目录固定为：
  - `frontend/`：前端代码、构建配置、离线依赖缓存配置
  - `backend/`：API、数据模型、服务层、调度器、自动化适配层
  - `deploy/`：Windows/Win7/Linux 部署脚本、服务模板、离线安装脚本
  - `docs/`：开发文档、接口文档、部署文档、联调文档、运维文档
- 不复用旧归档代码，按新边界重建。
- 所有模块从一开始按前后端分离设计，不保留服务端模板页方案。

#### 2. 前端架构
- 使用 `Vue3 + Vite + Vue Router + Pinia`。
- 路由分组固定：
  - `/auth/*`
  - `/admin/*`
  - `/member/*`
- 网络层统一：
  - `axios` 请求封装
  - 自动附带 access token
  - access token 过期时自动尝试 refresh
  - refresh 失败后跳回登录页
- 权限控制：
  - 路由守卫校验 `admin/member`
  - 页面级按钮权限控制
  - 越权页面跳转到 403/登录页
- Win7 兼容要求：
  - `build.target = "es2019"`
  - 必要 polyfill 固定纳入构建
  - 不使用依赖现代浏览器专属 API 的重型库
- UI 方向：
  - 中文后台风格
  - 管理端强调高密度信息展示
  - 成员端保持简洁只读

#### 3. 前端页面范围
- 认证：
  - 登录页

- 管理员端：
  - 看板页

  - 延期审批中心页
  - 用户管理页
  - 导入导出页
  - 审计日志页
- 成员端：
  - 我的任务列表
  - 我的任务详情
  - 我的通知列表
- 全局交互：
  - 删除任务、修改计划、手动改状态、延期审批、禁用用户、重置密码都必须二次确认
  - 成功/失败反馈统一样式
  - 全站中文

#### 4. 管理员看板页
- 指标卡片：
  - 任务总数
  - 进行中
  - 已完成
  - 已取消
  - 延期任务数
- 通知指标：
  - 邮件发送成功率
  - QAX 送达率
  - QAX 已读率
  - 失败重试数
- 图表：
  - 按负责人/成员完成数量
  - 按状态分布
  - 近 30 天完成趋势
  - 延期趋势
- 快捷入口：
  - 新建任务
  - 导入任务
  - 待审批延期
  - 通知中心
  - 用户管理

#### 5. 任务管理与里程碑
- 任务字段固定：
  - 标题
  - 内容
  - 负责人
  - 参与者
  - 开始时间
  - 结束时间
  - 优先级
  - 备注
- 任务状态：
  - 主状态：`未开始 / 进行中 / 已完成 / 已取消`
  - 子状态：`延期X天`
- 业务规则：
  - 创建任务时计算计划用时
  - 首次完成时计算实际用时
  - 管理员可手动改未完成任务计划
  - 管理员可删除任务，默认软删除并保留审计
  - 管理员手动改状态后可锁定状态
  - 锁定后邮件事件只记录，不自动覆盖状态
  - 管理员可解锁恢复邮件驱动
- 多里程碑：
  - 每任务多个节点
  - 节点可增删改排序
  - 每节点可配置提前 1-5 天提醒，多选
  - 节点时间必须落在任务开始/结束区间内

#### 6. 通知与模板体系
- 通知接收对象固定：
  - 任务负责人
  - 所有参与者
  - 自动去重
- 通知渠道：
  - `email`
  - `qax`
- 模板类型固定：
  - `MAIL_SEND`
  - `QAX_SEND`
  - `MAIL_REPLY`
- 发送模板：
  - 按通知类型绑定默认模板
  - 通知类型至少包括：任务创建、里程提醒、延期审批通知、手动提醒
  - 邮件发送模板正文必须包含回复模板指引
- 回复模板匹配规则：
  - 先主题后正文
  - 多模板命中按 `priority desc -> version desc -> template_id asc`
  - 每次命中保存 `resolved_template_id + resolved_version`
- 模板管理能力：
  - 新增/编辑
  - 版本化
  - 启用/停用
  - 设置默认
  - 冲突预览

#### 7. 邮件驱动业务
- 邮件能力包括：
  - 发送任务通知邮件
  - 轮询收件箱
  - 解析回复邮件
  - 驱动任务状态变更
  - 发起延期申请
  - 支持管理员邮件审批延期
- 邮件回复可触发：
  - 任务完成
  - 任务进行中
  - 延期申请
  - 延期审批
- 延期审批邮件解析必须满足：
  - 支持 `同意+新日期`
  - 支持 `拒绝+原因`
  - 支持中文冒号、英文冒号、空格混排
  - 日期支持 `YYYY-MM-DD`
  - 日期支持 `YYYY年M月D日`
  - 仅扫描前 20 行
  - 仅取首个匹配行
  - 忽略签名档、引用内容
- 失败处理：
  - `PARSE_FAILED`
  - `INVALID_DATE`
  - 不更新业务状态，进入人工处理队列

#### 8. QAX 自动化模块
- QAX 仅属于通知域，不回写任务主状态。
- 功能拆分：
  - `sender`：发送通知
  - `collector`：定时采集送达/已读结果
- 运行约束：
  - 默认仍按 Playwright 设计适配层
  - 若 Win7 运行受限，仅替换适配层实现，不改变 API 和业务模型
- 展示内容：
  - 通知是否发送成功
  - 是否送达
  - 是否已读
  - 失败原因
  - 重试次数
- 定时采集：
  - 配置化 cron/间隔
  - 前端仅展示采集结果，不暴露底层差异

#### 9. 延期申请与审批
- 发起路径：
  - 成员通过邮件回复命中延期申请模板
- 审批对象：
  - 所有管理员
- 审批通道：
  - Web 审批
  - 邮件审批
- 并发裁决规则：
  - 仅当 `approval_status = PENDING AND version = ?` 时可更新
  - Web 幂等键：`request_id + admin_id`
  - Mail 幂等键：`message_id + request_id`
  - 首个生效返回：`200 APPLIED`
  - 重放返回：`200 IDEMPOTENT_REPLAY`
  - 晚到返回：`409 ALREADY_DECIDED`
- 审批通过效果：
  - 更新任务截止时间/相关里程碑
  - 记录延期天数
  - 写延期事件流水
- 审批拒绝效果：
  - 保持当前计划
  - 写审批备注与拒绝原因

#### 10. 用户与角色管理
- 仅管理员可访问。
- 支持操作：
  - 新增用户
  - 编辑用户
  - 禁用/启用用户
  - 重置密码
  - 创建管理员/成员
- 数据约束：
  - `username` 唯一
  - `email` 唯一
  - `ip_address` 唯一
- 安全规则：
  - 禁止禁用最后一个管理员
  - 禁止把最后一个管理员降级为成员
  - 禁用用户后不可登录
- 密码策略：
  - 初始密码使用统一默认密码
  - 重置密码后恢复统一默认密码
  - 默认密码从配置项读取
- 审计要求：
  - 新增用户
  - 编辑用户
  - 禁用/启用
  - 重置密码
  - 角色变更
  - 全部写审计日志

#### 14. 部署与离线方案
- 方案 A：`venv + 启动脚本`
  - Windows/Win7：
    - Python 3.8
    - 前端构建产物 + 后端服务
    - NSSM 或 WinSW 托管
  - Linux：
    - Python 3.8+
    - systemd 托管
- 方案 B：Docker
  - Linux 标准部署方案
  - Windows 非 Win7 环境可选
  - Win7 不作为 Docker 主目标
- 方案 C：PyInstaller
  - 单机可执行分发包
  - 内含后端、静态资源、配置模板
- 离线能力：
  - Python wheel 仓
  - Node 离线依赖缓存
  - 一键安装脚本
  - 安装后环境校验脚本
- 内网部署文档必须包含：
  - 外网打包步骤
  - 内网导入步骤
  - 启动与回滚步骤
  - 常见故障定位

### Test Plan
1. 前端页面测试
- 管理员所有页面可访问且主要流程可用。
- 成员端页面只读，访问管理员页面时被正确拦截。
- 用户管理、任务管理、模板管理、通知中心、延期审批都能独立操作。

2. API 契约测试
- 所有接口使用 `/api/v1/*`。
- 登录、刷新、登出、未授权访问逻辑正确。
- 分页、过滤、错误码返回格式统一。
- DTO 字段与前端约定一致。

3. 用户与权限测试
- 管理员可新增成员和管理员。
- 成员调用管理员接口返回 `403`。
- 禁用用户后无法登录。
- 最后一个管理员不能禁用、不能降级。
- 重置密码后统一默认密码生效。

4. 任务与里程碑测试
- 任务创建、编辑、删除、状态修改完整可用。
- 多里程碑增删改排序正确。
- 节点提醒天数保存和读取正确。
- 手动锁定后邮件不覆盖状态。
- 解锁后邮件可继续驱动状态。

5. 模板与邮件测试
- 发送模板按通知类型绑定正确。
- 邮件正文包含回复模板指引。
- 多模板同时命中时结果稳定。
- 模板版本变化不影响历史事件回放。
- 完成、进行中、延期申请三类回复都能正确触发动作。

6. 延期审批专项测试
- Web 审批与邮件审批并发时仅一条生效。
- 同幂等键重复提交返回幂等成功。
- 晚到审批返回 `409 ALREADY_DECIDED`。
- “同意+新日期”支持：
  - 中文冒号
  - 英文冒号
  - 空格混排
  - `YYYY-MM-DD`
  - `YYYY年M月D日`
- 签名档和引用内容不干扰解析。
- 仅首个匹配行生效。

7. 通知与 QAX 测试
- 通知对象固定为负责人+参与者并自动去重。
- QAX 送达/已读状态正确展示。
- QAX 状态变化不影响任务主状态。
- 通知失败重试与错误展示正确。

8. 跨平台测试
- Windows 单机部署可以启动、登录、操作。
- Win7 环境核心流程可运行。
- Linux 环境运行行为一致。
- 离线安装包在内网环境可完成完整部署。

9. 文档验证测试
- Windows/Win7/Linux 三套部署文档可按步骤完成安装。
- 离线部署文档可独立完成外网打包与内网导入。
- 故障处理文档能覆盖常见启动失败、依赖缺失、数据库路径错误等问题。

10. 测试后数据库清理
- 所有自动化测试使用独立测试数据库路径，例如 `data/test.db`。
- 测试执行完成后必须执行“清空测试数据库”步骤：
  - 删除测试数据库文件
  - 删除测试运行产生的临时日志和缓存
  - 保留正式环境数据库和样例初始化数据不受影响
- 在测试脚本和文档中明确：
  - 测试库路径
  - 清库命令
  - 清理前校验条件
  - 清理后验证条件
- 若存在集成测试预置数据，清库后需再次校验测试环境为空库状态。

### Assumptions
- 主生产形态为 Windows 单机部署，Linux 为兼容部署路径。
- Win7 兼容目标为 Python 3.8 + ES2019 构建产物。
- V1 继续使用 SQLite，满足约 20 人规模。
- 默认密码使用配置项统一管理，后续可升级为首次登录强制改密。
- Playwright 优先作为 QAX 自动化适配层实现，如 Win7 实机验证存在兼容问题，仅替换适配层，不改变前后端契约。
