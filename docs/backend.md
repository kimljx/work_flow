# 后端说明

- 后端使用 FastAPI + SQLAlchemy + SQLite，接口统一挂在 `/api/v1`。
- 角色分为系统管理员、管理员和成员；系统管理员与管理员都属于管理角色并拥有管理接口权限，前端通过菜单可见性区分系统级配置入口。
- 任务创建、编辑、导入和邮件回执会共用同一套任务、成员、子任务和通知规则，避免不同入口出现行为漂移。
- 任务创建人必须按 `task.created_by` 直接查询用户，不能再从任务成员表反推。
- 通知服务会按接收人过滤子任务，确保不同成员收到的正文只包含分配给自己的子任务内容。
- 通知服务在渲染负责人与子任务时，直接按数据库查询任务成员和子任务，而不是依赖 `task.members`、`task.subtasks` 关系缓存，避免“任务刚创建时邮件正文为空”的问题。
- 通知服务新增了任务通知渲染预览能力，任务详情页与真实发送链路共用同一套模板与上下文解析逻辑，避免“页面预览正确、实际发送错误”的双轨偏差。
- 默认发送模板上下文统一补充 `task_content`、`task_remark`、`remind_focus`，默认邮件 / 即时消息正文都会带出主任务详情、主任务备注和当前提醒重点；若数据库中的默认模板仍是旧版正文，启动时会自动平滑升级，但不会覆盖管理员自定义过的模板。
- 任务提醒新增三种粒度：整任务提醒、子任务定向提醒、里程碑节点提醒。子任务提醒只发送给对应执行人，里程碑提醒面向当前任务成员广播节点重点。
- 通知接收人的 `read_status` 字段在当前版本按渠道解释：邮件渠道表示“是否已回复”，即时消息渠道表示“是否已读”；邮件收件链路识别到回复后，会回写最近一条匹配的邮件通知接收人记录，前端按渠道动态展示反馈口径。
- 模板新增与编辑接口会在入库前校验占位符，非法变量会直接拦截，避免错误模板进入“创建任务 -> 匹配模板 -> 发送通知”链路。
- 任务导入使用 Excel 模板，负责人、参与人员和子任务执行人统一按姓名匹配系统用户；遇到高重叠导入时，接口会要求前端二次确认。
- 删除任务时会同步清理该任务已生成的到期提醒通知记录，并将任务到期提醒配置置为关闭，避免定时扫描或通知中心继续出现已删除任务的提醒痕迹。
- 收件解析会优先读取 `text/plain`，若邮件仅提供 `text/html` 则自动转成纯文本并解码 `&nbsp;` 等实体；邮件详情同时保存完整原始正文，供前端按需弹窗查看。
- 系统发信会同时发送纯文本和 HTML alternative，HTML 版本会保留换行与空格，减少部分邮箱客户端把正文挤成一段的问题。
- SMTP 测试与发信链路现在会单独识别 SSL/TLS 握手异常；遇到 `_ssl.c:1007`、`wrong version number` 这类报错时，会直接提示检查 `25 / 465 / 587` 端口与 `SMTP_USE_SSL`、`SMTP_USE_TLS` 的匹配关系，而不是只返回底层异常。
- IMAP 测试与自动收件链路现在支持明文、STARTTLS、SSL 三种连接方式；当收信服务器不使用 SSL 时，可通过 `IMAP_USE_SSL=false` 切换，而不需要修改业务代码。
- 自动收件现在支持 `MAIL_INBOX_PROTOCOL=imap|pop3` 两种协议；当邮箱只提供 `POP3 110/995` 时，可直接切换到 POP3，并沿用基线与去重逻辑避免重复入库。
